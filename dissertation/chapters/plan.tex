\section{Overview}
This chapter describes how the Agentic AI Tutor was implemented following the MLOps lifecycle. It explains how each stage of the pipeline was carried out in practice.

The development of the Agentic AI Tutor is organised into six main stages that follow the MLOps lifecycle:
\begin{itemize}
    \item Data collection and processing
    \item Model development and training
    \item Model evaluation and testing
    \item Model deployment
    \item Monitoring and maintenance
    \item Lifecycle management
\end{itemize}

\section{Data collection and processing}
\subsection{Data collection}
The Agentic AI Tutor focuses on three modules:

\begin{itemize}
    \item \textbf{Fundamentals of Software Development (FSD)},
    \item \textbf{Fundamentals of Computer Science (FCS)},
    \item \textbf{Discrete Mathematics (DMA)}.
\end{itemize}

The Agentic AI Tutor targets three academic modules: Fundamentals of Software Development (FSD),
Fundamentals of Computer Science (FCS), and Discrete Mathematics (DMA). In this project, the full data processing, training, and evaluation pipeline was implemented and applied consistently to all three subjects. Each subject was processed independently using the same subject-agnostic pipeline, resulting in separate datasets, fine-tuned models, and evaluation results for FSD, FCS, and DMA.

\paragraph{Sources of teaching materials.}
All teaching materials were downloaded from the official university Moodle platform.
For FSD, the 2022/23 Moodle course was used.

\begin{itemize}
    \item Fundamentals of Software Development (CSI\_4\_FSD\_2223)
\end{itemize}

The FSD course for 2024/25 is not visible to students on Moodle, and the 2025/26 course does not yet contain a full set of lecture and tutorial resources. The 2022/23 course was therefore chosen because it provides a complete set of materials for the whole module.

For FCS and DMA, the raw resources were downloaded from the 2024/25 Moodle courses:

\begin{itemize}
    \item Fundamentals of Computer Science (CSI\_4\_FCS\_2425)
    \item Discrete Mathematics (CSI\_4\_DMA\_2425)
\end{itemize}

\paragraph{Directory structure and file formats.}
All teaching resources are stored in a data/ directory in the project repository. Each subject has its own subdirectory with a raw/ folder that contains the original files as downloaded from Moodle:

\begin{verbatim}
data/
  fsd/
    raw/
  fcs/
    raw/
  dma/
    raw/
\end{verbatim}

This structure makes it easy to reproduce the experiments. Any script that needs the materials can read all files in data/<subject>/raw/.

For FSD, all files are in PDF format. The fsd/raw directory contains lecture slide decks, tutorial sheets, mixed learning session slides, one project handout, and tutorial solution documents.

For FCS and DMA, the raw directories contain a mix of PPTX slide decks, DOC/DOCX tutorial sheets and solutions, and several supporting PDFs (for example, short chapters or summary notes).

The data loading code used later in the project can read PDF, PPTX, DOC, DOCX, TXT and Markdown (MD) files. This means that all teaching materials from Moodle can be processed by the same pipeline.

\subsection{Data exploration}
Before any downstream steps, an exploratory analysis of the teaching materials for the three target modules (FSD, FCS and DMA) was carried out. To make automatic grouping possible, all files were manually renamed so that the filename starts with a topic prefix in the form Topic\_XX. This makes it possible to group files by topic in an automated way.

\paragraph{Building topic-level documents.}
In the pipeline, the main unit is a topic document. For each subject, all files that share the same Topic\_XX prefix are grouped together. Text is extracted from each file (PDF, PPTX, DOC/DOCX), and all extracted text within the same topic is concatenated into one topic-level document (for example, FSD\_Topic\_03).

\paragraph{Normalising input into topic texts.}
All subject materials (PDF slides, PPTX lecture decks, and Word-based tutorials) were first consolidated into a unified intermediate representation stored as a topics.pkl file. Each entry is indexed by (subject, topic\_id) and contains the extracted text (full\_text) together with lightweight provenance metadata (a list of source\_files). This normalisation enables the same QA generation pipeline to be reused across subjects without relying on format-specific logic.

\subsection{Data preparation and preprocessing}

The teaching materials were transformed into a question-answer dataset suitable for training a small language model as an AI tutor. While the pipeline itself is subject-agnostic, model fine-tuning was performed separately for each subject. Three independent LoRA adapters were trained, one per subject, each using only the training data from its corresponding module.

\paragraph{Token-based chunking of long topics.}
Topic texts vary substantially in length. Tutorial sheets and consolidated notes can exceed the practical prompt budget for a small model. Therefore, each topic was split into overlapping chunks using the model tokenizer, with a target size of 1200 tokens and an overlap of 120 tokens. Chunks were limited to 1200 tokens to keep the full prompt (instructions + context) and the generated answer within the model’s 4096 token context window, while avoiding aggressive truncation and preserving space for multi-sentence tutor-style explanations. Chunking is performed on paragraph boundaries when possible, and oversized paragraphs are further split by sentence boundaries to avoid extremely long single segments. This step ensures that the model receives coherent context while leaving sufficient generation budget for explanatory answers.

\paragraph{Tutor-oriented QA generation.}
Question–answer pairs were generated using the microsoft/Phi-3-mini-4k-instruct model. The prompt was designed for a tutoring setting: answers are expected to be explanatory (not one-line definitions), typically spanning multiple sentences and focusing on understanding rather than rote memorisation. To increase coverage, the pipeline generates multiple question types per chunk using universal modes: core concepts, definitions and terms, application and reasoning, compare/contrast, and common mistakes. Since Phi-3-mini supports a 4k context window, the generation prompts were designed to fit within this budget (instructions + chunk context + answer). The target number of questions per chunk is chosen heuristically based on chunk length (4 for chunks under 300 tokens, 8 for 300-699 tokens, and 12 for chunks of over 700 tokens).

For each subject, the generated QA dataset was first split into training and evaluation subsets.
This split was performed before any automated or manual quality review to prevent information leakage between training and evaluation data. All subsequent processing steps, including GPT-based quality review and manual inspection, were applied independently to the training and evaluation splits.

\paragraph{Quality filtering.}
Because the target model is trained as an AI tutor, the dataset prioritises explanatory answers rather than one-line definitions. After generation, each candidate QA object is validated and filtered in several stages:

\begin{enumerate}
    \item \textbf{JSON validation and repair.} The model is instructed to output only a JSON array. The pipeline attempts strict JSON parsing. If parsing fails, it issues one repair prompt requesting JSON-only output. Outputs that remain unparsable after the retry are logged in qa\_errors\_<subject>.csv.
    \item \textbf{Answer and question length filters.} Only QA pairs passing a deterministic quality function are retained. In particular, questions must contain at least 4 words and answers must contain at least 20 words to enforce tutor-style explanations.
    \item \textbf{Deduplication.} To reduce redundancy within each subject, questions are normalised (lowercasing, whitespace normalisation, and punctuation removal) and normalised exact duplicates are removed. In the final datasets, the number of duplicate questions after normalisation is 0 for FSD, FCS and DMA.
    \item \textbf{Semi-automated quality review.} All generated QA pairs were reviewed using an automated GPT-based reviewer, which assigned each pair a label (OK or FLAGGED) together with optional reasons. To validate the reliability of this filtering step, random samples from both the OK and FLAGGED subsets were manually inspected. This inspection confirmed that the automated review reliably retained high-quality tutor-style explanations while filtering out low-quality or unsuitable examples.
    \item
\end{enumerate}

\paragraph{Evaluation subset used for metrics.}
For final metric reporting, only the evaluation subset labelled OK (saved as <subject>\_eval\_OK.jsonl) was used. This ensures that all reported scores are computed on high-quality tutor-style QA pairs and remain strictly held out from training.

\paragraph{Resulting QA datasets.}
The final datasets are stored as JSONL files (one QA pair per line) including metadata fields such as subject, topic\_id, chunk\_id, mode, and source\_files. The resulting dataset sizes are:
\begin{itemize}
    \item \textbf{FSD:} 680 QA pairs from 16 topics and 60 chunks (59 chunks used at least once), covering 31 source files.
    \item \textbf{FCS:} 745 QA pairs from 10 topics and 73 chunks (69 chunks used at least once), covering 32 source files.
    \item \textbf{DMA:} 708 QA pairs from 10 topics and 74 chunks (72 chunks used at least once), covering 25 source files.
\end{itemize}

\paragraph{Conversion to conversational fine-tuning format.}
For LoRA fine-tuning with Unsloth, the simple QA pairs are converted into a conversational format. A separate script reads the flat JSONL file with question-answer pairs and wraps each pair into a short conversation: a system message defining the tutor role for the subject, followed by the student question (user message) and the tutor answer (assistant message).

\section{Model development and training}
Although Phi-3-mini-4k-instruct was used for synthetic QA generation, model fine-tuning and evaluation were performed using microsoft/Phi-4-mini-instruct as the base model, with one LoRA adapter trained per subject (FSD, FCS, DMA). This keeps the training/evaluation stack consistent with the final inference setup used for metric reporting.


Model adaptation was performed separately for each subject. Starting from the same instruction-tuned
base model, three independent LoRA fine-tuning runs were executed, resulting in subject-specific
adapters for FSD, FCS, and DMA. This design choice allows each model to specialise in the terminology, problem types, and explanatory style of its corresponding module, while preserving a consistent training and evaluation methodology across subjects.

The goal of the Agentic AI Tutor is to adapt an existing instruction-tuned model to a specific academic tutoring domain. Low-Rank Adaptation (LoRA) was selected as the main fine-tuning method. The implementation follows best practices described in the Unsloth documentation \parencite{LoRAfine55:online} and uses the Unsloth training framework and example notebooks \parencite{GitHubun10:online} as a reference for stable and efficient fine-tuning.

\subsection{LoRA configuration}

The LoRA configuration was chosen based on recommendations from the Unsloth hyperparameter guide \parencite{LoRAfine55:online} and preliminary experiments. The following key settings were used:

\begin{itemize}
    \item \textbf{LoRA rank ($r=16$).} This value provides sufficient capacity for domain adaptation while keeping the number of trainable parameters low.
    \item \textbf{LoRA scaling factor ($\alpha=16$).} Setting $\alpha$ equal to the rank follows a commonly recommended heuristic and results in stable training behaviour.
    \item \textbf{Target modules.} LoRA adapters were applied to all major linear layers in both the attention and feed-forward blocks (query, key, value, output, and MLP projections). This ensures that the model can adapt both its attention patterns and its internal representations.
    \item \textbf{LoRA dropout ($0.0$).} Dropout was disabled, as the fine-tuning runs are short and the dataset is already filtered for quality.
    \item \textbf{Bias training.} Bias terms were not trained to reduce unnecessary parameter updates.
\end{itemize}

Gradient checkpointing provided by Unsloth was enabled to further reduce memory consumption during training.

\subsection{Training setup}

The training configuration was designed to achieve a stable effective batch size while remaining within GPU memory limits. The main parameters are:

\begin{itemize}
    \item \textbf{Per-device batch size:} 2
    \item \textbf{Gradient accumulation steps:} 8
    \item \textbf{Effective batch size:} 16
    \item \textbf{Learning rate:} $2 \times 10^{-4}$
    \item \textbf{Weight decay:} 0.01
    \item \textbf{Warm-up ratio:} 5\%
    \item \textbf{Number of epochs:} 1
\end{itemize}

An effective batch size of 16 is recommended in the Unsloth documentation as a good default for stable fine-tuning of instruction-based datasets. Training for a single epoch was chosen to reduce the risk of overfitting, as the dataset consists of synthetic but carefully filtered tutor-style question-answer pairs.

\subsection{Design choice: training without response-only masking}

Some fine-tuning pipelines apply loss masking so that the model is trained only on assistant tokens, ignoring system and user messages. While this technique can be useful for large multi-turn conversational datasets, it was deliberately not used in this project.

During implementation, applying response-only masking caused all training labels to be masked out due to strict token-level matching requirements in the training framework. More importantly, the dataset used in this project consists exclusively of single-turn tutor-style interactions with a clear and consistent structure. In this setting, training on the full input sequence does not introduce significant noise and simplifies the training pipeline.

For these reasons, the final implementation keeps the training code unchanged and performs standard supervised fine-tuning without response-only masking. This decision improves robustness and reproducibility while preserving the intended tutoring behaviour of the model.

\subsection{Training environment}

All fine-tuning experiments were conducted in Google Colab using a free NVIDIA Tesla T4 GPU. The Unsloth framework was used to optimise memory usage and training speed.

The trained model is saved as LoRA adapter weights. This allows the adapters to be reused with the original base model during inference, or merged into a standalone model if required for deployment.

\subsection{Retrieval-Augmented Generation}

In addition to fine-tuning the base model with LoRA adapters, the Agentic AI Tutor supports retrieval-augmented generation (RAG) to ground its answers in the original teaching materials.

The RAG pipeline reuses the same token-based chunks created during the data preparation stage. Instead of treating full lecture or tutorial files as retrieval units, each subject corpus is indexed at the chunk level. This design choice is motivated by a factor that semantic retrieval performs better when embeddings represent short, topically coherent text segments rather than long documents containing multiple concepts.

Each retrieved chunk retains lightweight provenance metadata, including the topic identifier and source file names. This allows the system to trace answers back to specific lecture slides or tutorial documents, which is useful for debugging, evaluation, and future user-facing citation features.

For each subject (FSD, FCS and DMA), all chunks are embedded using a lightweight sentence-transformer model and stored in a persistent vector database (ChromaDB). The RAG component operates independently of fine-tuning.

\section{Model evaluation and testing}

Model quality was evaluated separately for each subject (FSD, FCS, DMA) using the corresponding eval\_OK subset. This evaluation set contains only question-answer pairs that passed the semi-automated GPT-based quality review and subsequent manual spot-checking.

We compared five model variants:
\begin{itemize}
    \item \textbf{BASE}: the instruction-tuned base model without adaptation,
    \item \textbf{LoRA}: the base model with a subject-specific LoRA adapter,
    \item \textbf{LoRA+RAG}: the LoRA-adapted model augmented with retrieval of subject materials,
    \item \textbf{Qwen3-32B}: an external LLM baseline,
    \item \textbf{Llama-70B}: an external LLM baseline.
\end{itemize}

All metrics were computed within each subject (FSD/FCS/DMA) using the same evaluation script and the same prompt format. Each model answer was compared to the held-out reference tutor answer.

\subsection{Metrics}
To measure answer quality, we use both lexical overlap metrics (how much the text matches the reference wording) and a semantic similarity metric (how similar the meaning is even if the wording changes). We also report runtime (latency). Lexical-overlap metrics (BLEU, ROUGE-L, METEOR) reward shared words and phrases, so they are sensitive to paraphrasing. Semantic similarity (BERTScore) is more robust to paraphrase. It can increase even if lexical overlap stays modest.

\paragraph{BLEU (lexical overlap).}
BLEU measures modified n-gram precision between a candidate answer and a reference answer, with a brevity penalty for very short outputs \parencite{BleuaMet4:online}. In this project, BLEU is reported on a 0-100 scale. In tutoring-style answers, BLEU is usually low because there are many correct ways to explain the same idea.

\paragraph{ROUGE-L (lexical overlap).}
ROUGE-L is based on the longest common subsequence (LCS) between candidate and reference \parencite{ROUGEAPa16:online}. It is reported in the 0-1 range: 1 means identical text order, 0 means no overlap in sequence.

\paragraph{METEOR (lexical overlap with flexible matching).}
METEOR aligns candidate and reference using unigram matches (including stemming and synonymy) and combines precision and recall with a fragmentation penalty \parencite{METEORAn22:online}. It is also reported in the 0-1 range.

\paragraph{BERTScore F1 (semantic similarity).}
BERTScore compares candidate and reference using contextual embeddings and reports F1 similarity \parencite{Zhang2019BERTScore}. Higher values mean the meaning is closer even if words differ. 

\paragraph{Latency.}
Latency is reported as mean inference time per example (seconds). Lower is better.

\subsection{Analysis}

BLEU and ROUGE-L are lexical-overlap metrics. They reward using the same words and similar word order as the single reference tutor answer. BLEU is reported on a 0-100 scale (here mostly 5-12 for SLMs), so low absolute values mainly mean limited exact phrase overlap, which is normal when many correct tutor-style phrasings exist. ROUGE-L (0-1) measures the longest common subsequence. Values around 0.13-0.22 indicate modest in-sequence overlap for multi-sentence explanations. METEOR (0-1) uses unigram matching (including flexible matching such as stemming/synonymy) and adds a penalty when matched words appear in fragmented order. Values around 0.24-0.30 here mean partial alignment to the reference. BERTScore F1 is a semantic-similarity metric. Higher values indicate stronger meaning similarity even when surface wording differs. In this project BERTScore is reported in a rescaled form.

\paragraph{BASE vs.\ LoRA (effect of fine-tuning).}
Figure \ref{fig:all-bleu} and Figure \ref{fig:all-bertscore} summarise the main improvements from BASE to LoRA across subjects.

\begin{figure}[h]

Across all three subjects, LoRA improves BLEU-2, ROUGE-L, and BERTScore compared to BASE:
\begin{itemize}

These gains mean the LoRA adapters help the SLM follow subject terminology and produce explanations that are closer to the reference answers both in surface form (BLEU-2/ROUGE-L) and in meaning (BERTScore).

\subsection{Key findings}
LoRA fine-tuning is the main reason for quality improvement across FSD, FCS, and DMA. Compared to BASE (Phi-4-mini-instruct, 3.8B parameters), LoRA consistently increases BLEU-2, ROUGE-L, and BERTScore F1. This means the SLM becomes closer to the reference tutor answers both in wording (lexical overlap) and in meaning (semantic similarity). These improvements support the project aim: small, subject-specific models can be adapted to produce more tutor-like explanations without training a large model from scratch.

RAG gives smaller and less consistent gains than LoRA. In FSD and DMA, LoRA+RAG improves BLEU-2 and ROUGE-L further, while in FCS the effect is mixed (BLEU-2 slightly increases, but ROUGE-L is almost unchanged). This is expected when evaluation uses only one reference answer: retrieval can add correct details or change phrasing, which may help tutoring quality but does not always increase overlap with a single reference.

METEOR often decreases after LoRA/RAG. A likely reason is that adapted models use more domain-specific wording and a different sentence structure, which reduces word-level alignment and increases fragmentation relative to the single reference, even when answers remain correct and clear.

When comparing to the LLM baselines (Llama-3.3-70B and Qwen3-32B), the picture depends on the metric. The best SLM variants are already competitive and in this evaluation they outperform both LLMs on BLEU-2 and BERTScore F1 in all three subjects. At the same time, Llama generally remains higher on ROUGE-L and METEOR, which suggests closer word-order alignment to the reference. 

The efficiency results show a different trade-off. Among the deployed SLM variants, LoRA usually achieves the lowest mean latency, while LoRA+RAG remains close behind and only slightly increases runtime because of the retrieval step. Although BASE has the highest token throughput, it also generates substantially longer outputs on average, which leads to higher end-to-end latency. Qwen3-32B is the slowest model among those with available runtime measurements. Overall, the subject-specific LoRA-based deployment provides a favourable quality-latency trade-off, with LoRA+RAG offering better answer quality at only a modest additional runtime cost.

\section{Model deployment}

The Agentic AI Tutor is deployed as a containerised microservices system orchestrated by Kubernetes. The deployment consists of subject-specific backend services, a frontend web application, and dedicated ChromaDB instances for retrieval, all managed within the same cluster. This section describes the overall deployment architecture, the backend and frontend services, the containerisation approach, and the Kubernetes orchestration layer.

\subsection{System architecture overview}

The deployed system follows a microservices pattern in which each academic subject is served by an independent backend container. This design directly reflects the methodology requirement of isolated, subject-specific modules that can be developed, evaluated, and scaled independently.

\begin{figure}[h]
\centering
\includegraphics[width=\linewidth]{figs/Deployment-Pipeline.png}
\caption{Deployment view of the Agentic AI Tutor on Kubernetes.}
\label{fig:deployment}
\end{figure}
\FloatBarrier

As shown in Figure \ref{fig:deployment}, the system consists of four main layers:

\begin{itemize}
    \item \textbf{Ingress layer.} An Ingress controller routes external HTTP requests to either the frontend service (path /) or to a subject-specific backend service (paths /api/fsd, /api/fcs, /api/dma). Each backend exposes the same inference endpoint (/api/chat). The Ingress path prefix selects which subject service receives the request.
    \item \textbf{Frontend.} A Next.js application that provides the chat interface. It communicates with the backend through the Ingress, sending requests to /api/chat with the selected subject included in the JSON body.
    \item \textbf{Backend microservices.} Three FastAPI-based backend services are deployed from the same container image, with each service dedicated to a single subject. The subject-specific LoRA adapter is embedded in the container image, while retrieval data is stored in an external ChromaDB instance deployed separately inside the Kubernetes cluster. The SUBJECT environment variable selects which subject configuration is activated at runtime. Each backend service is configured for subject-specific runtime isolation. GPU scheduling is environment-dependent and not hard-enforced in the current Kubernetes manifests.
\end{itemize}

This separation improves fault isolation, since a failure in one subject-specific service does not directly interrupt the others. Each service can be restarted, updated, and scaled independently. In Kubernetes, each service is managed by its own Deployment, which maintains multiple underlying pod instances, automatically restarts failed replicas, and supports scaling up or down in response to demand. In addition, retrieval storage is externalised into a dedicated ChromaDB instance for each subject. For that reason the application layer remains stateless at runtime, which simplifies dynamic scaling. Because each subject has its own Deployment, database, and embedded LoRA adapter, the services can be scaled independently according to subject-specific traffic and resource requirements.

\subsection{Backend service}

Each backend microservice is a FastAPI application that combines a LoRA-adapted language model with a retrieval-augmented generation (RAG) pipeline. The same Docker image is used for all three subjects. The subject configuration file subjects.yaml and the subject-specific LoRA adapters are baked into the backend image during the Docker build. At runtime, the SUBJECT environment variable selects which subject configuration and embedded LoRA adapter to activate, and determines which external ChromaDB instance inside the Kubernetes cluster the service connects to.

\paragraph{Model loading pipeline.}
When a container starts, the application performs the following initialisation sequence:
\begin{enumerate}
    \item Load subject configuration from subjects.yaml.
    \item Initialise the sentence-transformer embedding model.
    \item Load the tokenizer for Phi-4-mini-instruct.
    \item Load the base model with 4-bit quantisation.
    \item Attach the subject-specific LoRA adapter using the PEFT library.
    \item Connect to the persistent ChromaDB vector store for the selected subject.
\end{enumerate}

After initialisation, the service exposes a /health endpoint that reports readiness (model and index loaded) and the active subject identifier, which is used by Kubernetes readiness probes.

\paragraph{Inference flow.}
When a user submits a question, the backend processes it through a retrieval-then-generation pipeline.

\begin{figure}[h]
\centering
\includegraphics[width=\linewidth]{figs/Query-Processing.png}
\caption{Sequence diagram of the inference request.}
\label{fig:inference-sequence}
\end{figure}
\FloatBarrier

The retrieval-augmented inference pipeline consists of four stages:

\begin{enumerate}
    \item \textbf{Retrieval.} The user question is embedded using the sentence-transformer model and matched against the subject's ChromaDB collection. The top-6 most similar chunks are returned along with their similarity scores and provenance metadata (topic, source file).
    \item \textbf{Context building.} The retrieved chunks are concatenated into a context string and truncated to a maximum of 1\,400 tokens to leave room for the system prompt and the generated answer within the deployed context budget.
    \item \textbf{Generation.} The context, system prompt, and user question are formatted into the Phi-4 chat template and passed to the LoRA-adapted model. Generation uses greedy decoding and a maximum of 192 new tokens, with stop markers to prevent runaway generation.
    \item \textbf{Response formatting.} The generated text is returned together with the top-3 most relevant source chunks (selected from the retrieved top-6) and their similarity scores, allowing the frontend to display compact information.
\end{enumerate}

The service also provides a streaming endpoint (/api/chat/stream) that yields tokens incrementally, reducing perceived latency for the end user.

\subsection{Frontend service}

The frontend is a Next.js application (version 16) that provides a chat-based interface for interacting with the tutor. It is built as a standalone production bundle, which produces a self-contained server.js file suitable for containerised deployment.

\paragraph{Mode routing.}
The frontend supports microservices mode. Separate variables (NEXT\_PUBLIC\_FSD\_API\_URL, NEXT\_PUBLIC\_FCS\_API\_URL, NEXT\_PUBLIC\_DMA\_API\_URL) point to individual backend services. In Kubernetes, these URLs resolve via the Ingress controller (using per-subject path prefixes) or via internal service DNS names depending on the deployment configuration.

\paragraph{User interface.}
The interface is built with React 19 and Tailwind CSS. It provides subject selection tabs, a conversational chat panel, and source cards that display retrieved chunks with their similarity scores. The frontend uses Axios for API communication and currently uses synchronous chat requests (/api/chat) in the deployed UI. A streaming backend endpoint is available for future frontend integration.

\subsection{Talking-head avatar}

To provide a more engaging and accessible interaction mode, the system includes a talking-head avatar feature. The system renders a 3D avatar in the browser and animates it in real time. The response text is converted to speech, and the corresponding phoneme and viseme data are used to synchronise the avatar’s lip movements. This produces a responsive, interactive multimodal output that combines audio and animation.

\paragraph{Pipeline.}
The avatar pipeline consists of two tightly integrated stages:

\begin{enumerate}
    \item \textbf{Speech synthesis.} The response text is processed using HeadTTS, a browser-based text-to-speech engine. In addition to audio output, HeadTTS provides phoneme-level timing and viseme data required for animation.
    \item \textbf{Avatar rendering and animation.} The audio and viseme stream are passed to the TalkingHead renderer, which animates a 3D avatar directly in the browser. Lip movements are driven in real time using the viseme data, allowing the avatar to synchronise with speech without generating intermediate video files.
\end{enumerate}

The pipeline operates entirely on the client side, eliminating the need for backend video generation endpoints. This reduces system complexity and enables low-latency interaction.

\paragraph{TalkingHead integration.}
TalkingHead is integrated as a client-side rendering component within the frontend application. It loads a predefined 3D avatar model and applies animation updates based on incoming viseme data.

Unlike video-based approaches, no pre-processing of images or generation of intermediate assets is required. The avatar remains persistent in the interface and updates dynamically as speech is produced. This allows continuous interaction rather than discrete video playback.

Because both speech synthesis and animation are executed in the browser, the system avoids GPU-bound backend services and simplifies deployment. However, this approach depends on client-side performance and browser capabilities.

\subsection{Containerisation}

\paragraph{Backend Dockerfile.}
The backend image is based on pytorch/pytorch:2.2.0-cuda12.1-cudnn8-runtime, which includes the CUDA runtime needed for 4-bit quantisation via bitsandbytes. The image installs Python dependencies from requirements.txt, copies the application source code, subjects.yaml, and the subject-specific LoRA adapters, and runs Uvicorn on port 8000. The LoRA adapters are therefore baked into the image, whereas retrieval storage is externalised to dedicated ChromaDB instances running separately inside the Kubernetes cluster.

\paragraph{Frontend Dockerfile.}
The frontend uses a multi-stage build to minimise the final image size:

\begin{enumerate}
    \item \textbf{deps stage}: installs Node.js dependencies with npm ci on the node:20-alpine base image.
    \item \textbf{builder stage}: copies the source code and dependencies, sets API URL build arguments, and runs npm run build to produce the Next.js standalone output.
    \item \textbf{runner stage}: copies only the standalone server, static assets, and public files into a clean Alpine image. A non-root user (nextjs, UID 1001) is created for security.
\end{enumerate}

The final image exposes port 3000 and starts the Node.js server directly.

\paragraph{Docker Compose mode.}

{Microservices mode} (docker-compose.microservices.yml) runs three backend containers (FSD on port 8001, FCS on port 8002, DMA on port 8003), each configured with a SUBJECT environment variable and a dedicated GPU. The frontend is built with per-subject API URLs. This mode requires three GPUs and provides fault isolation between subjects.

Microservices mode configure health checks that poll the /health endpoint every 30 seconds. Data volumes are not mounted to the backend containers. Instead, each backend service connects to its dedicated ChromaDB container via HTTP using the CHROMA\_HOST and CHROMA\_PORT environment variables, and ChromaDB persistence is managed through named Docker volumes.

Each subject (FSD, FCS, DMA) has its own Kubernetes Deployment with the following configuration:

\begin{itemize}
    \item \textbf{Replicas:} 1 per subject in this deployment (in the current manifests, replicas are set to 1 per subject, scaling policy can be adjusted per environment).
    \item \textbf{Resource requests:} 4\,Gi memory, 1 CPU.
    \item \textbf{Resource limits:} 12\,Gi memory, 4 CPUs.
    \item \textbf{Node placement:} no dedicated GPU nodeSelector/toleration is enforced in the current manifests, scheduling remains environment-dependent.
    \item \textbf{Health probes:} readiness probe on /health (initial delay 300\,s, period 10\,s), liveness probe on /health (initial delay 600\,s, period 30\,s). The longer liveness delay accounts for the time needed to load the quantised model and LoRA adapter into GPU memory.

\end{itemize}

Each Deployment is fronted by a ClusterIP Service that provides a stable internal DNS name (e.g.\ fsd-service.ai-tutor.svc.cluster.local).

\paragraph{Frontend Deployment.}
The frontend runs with 2 replicas and lighter resource requirements (256\,Mi--512\,Mi memory, 100\,m--500\,m CPU). Its environment variables are set to the Ingress endpoint (and per-subject routing prefixes) or to Kubernetes-internal DNS names depending on the chosen routing configuration.

\paragraph{Ingress routing.}
An Ingress resource maps external HTTP paths to internal services:

\begin{itemize}
    \item / $\rightarrow$ frontend service (port 80)
    \item /api/fsd $\rightarrow$ fsd-service (port 80)
    \item /api/fcs $\rightarrow$ fcs-service (port 80)
    \item /api/dma $\rightarrow$ dma-service (port 80)
\end{itemize}

The manifests include annotations for both AWS ALB and NGINX Ingress controllers, making the configuration portable across cloud providers.

\paragraph{Kustomize management.}
All Kubernetes manifests are managed through a kustomization.yaml file that declares the full set of resources (namespace, storage, per-subject deployments and services, frontend deployment and service, and ingress). This allows the entire stack to be applied with a single kubectl apply -k k8s/ command and ensures that the deployed state matches the version-controlled manifests.


\section{Monitoring and maintenance}
\subsection{Operational logging}

The backend service uses Python's standard logging module for structured operational visibility. Logging is configured centrally in logging\_config.py, which is imported at application startup before any other module is initialised. The configuration sets a uniform log format across all modules.

The log level is controlled at runtime through the LOG\_LEVEL environment variable (default INFO), so verbosity can be adjusted per deployment without rebuilding the image. Noisy third-party loggers (transformers, chromadb, sentence\_transformers, uvicorn.access) are suppressed to WARNING level to keep the output focused on application events.

\paragraph{Startup logging.}
During initialisation, each step of the model loading pipeline produces an INFO-level message. This makes it easy to identify slow startup or failures during adapter loading or ChromaDB connection setup, which is especially useful given the multi-step initialisation sequence and the long model loading time.

\paragraph{Per-request inference logging.}
After every generation call, the inference module logs the subject identifier, the elapsed wall-clock time, and the number of tokens produced. This gives a lightweight per-request latency record without requiring an external metrics system. The log entries can be collected from container stdout to track response-time trends over time.

\paragraph{RAG retrieval logging.}
Each retrieval call logs the subject, the number of chunks returned, and a short prefix of the query at INFO level. At DEBUG level, the similarity score and a brief text preview are logged for the top three retrieved chunks. Connection failures and missing ChromaDB collections are reported at WARNING and ERROR levels respectively, which supports quick diagnosis of retrieval problems.


\section{Lifecycle management}

\subsection{CI/CD pipeline}

A continuous integration and delivery pipeline was implemented using GitHub Actions and Argo CD. The GitHub Actions workflow is organised into six jobs: lint, build, build-frontend, playwright, scan, and gitops-update. The backend and frontend build jobs depend on lint. The Trivy-based scan job depends on the backend build. Gitops-update depends on both build jobs. The playwright job runs separately as part of the same workflow to validate frontend behaviour through smoke tests.

Figure \ref{fig:github-actions} illustrates the GitHub Actions workflow used in the project. The pipeline consists of multiple stages, including syntax validation, container image build and push, frontend testing, and vulnerability scanning, with dependencies between jobs explicitly defined.

    \centering

    \caption{GitHub Actions workflow showing CI/CD pipeline with syntax checking, container image build and push, frontend testing, and vulnerability scanning stages.}
    \label{fig:github-actions}
\end{figure}
\FloatBarrier

The lint job runs on a standard Ubuntu runner without a GPU. It first checks the Python syntax of all backend modules using py\_compile, covering both the main service (config.py, rag.py, inference.py, main.py, schemas.py, logging\_config.py). It then validates the Kubernetes manifests for structural correctness using kubeconform. This job runs on every push and every pull request, providing fast feedback before a Docker image is built.


\paragraph{Security scanning.}
The lint job also includes two automated security checks. First, Bandit, a static analysis tool for Python, scans the backend source code for common security issues. Second, the frontend dependencies are checked with npm audit at high severity level, so the audit focuses on high and critical vulnerabilities in the dependency tree.

On pushes to the main branch, two dedicated jobs build and push images to GHCR: the build job for the backend image and the build-frontend job for the frontend image. For each image, a latest tag and a commit-SHA tag are produced (for example, ghcr.io/nasttolm/ai-tutor-backend:<sha> and ghcr.io/nasttolm/ai-tutor-frontend:<sha>) that uniquely identifies each build. Build-layer caching through the GitHub Actions cache is enabled to reduce image build time on repeated pushes.

\paragraph{Container image scanning.}
The scan job runs after the build job and checks the published Docker image for vulnerabilities using Trivy \parencite{trivy}. While Bandit analyses the source code, Trivy examines the built container image itself, including operating system packages, system libraries, and installed Python dependencies. The scan focuses on HIGH and CRITICAL severity issues. The results are exported in SARIF format and uploaded to the GitHub Security tab through the CodeQL upload action. This makes the vulnerabilities visible in the repository’s security dashboard, but it does not stop the pipeline because the scan uses exit-code: 0. This separation is useful because it covers different layers of risk: source-level problems can be identified early in the lint job, while supply-chain and base-image vulnerabilities are reported after the image has been built.

\subsection{GitOps deployment with Argo CD}

Deployment to Kubernetes is managed with Argo CD following the GitOps approach. Two manifests are provided in k8s/argo/: an AppProject that restricts deployments to the ai-tutor namespace from the project repository, and an Application that declares the desired cluster state.

The Application manifest points Argo CD at the k8s/ directory of the repository and instructs it to render the manifests using Kustomize. Two synchronisation policies are enabled: prune, which removes cluster resources that have been deleted from the repository, and selfHeal, which reverts any manual changes made directly to the cluster. Together, these policies make the Git repository the single source of truth for the cluster state.

The end-to-end delivery flow works as follows:
\begin{itemize}
    \item A developer pushes a commit to main; GitHub Actions builds and pushes a new image tagged with the commit SHA.
    \item The gitops-update job automatically updates backend and frontend image tags in k8s/kustomization.yaml and commits the change to the repository.
    \item Argo CD detects the change during its next polling cycle and rolls out the updated Deployment to the cluster.
\end{itemize}

Figure \ref{fig:argo-cd} shows the Argo CD dashboard for the deployed application. The interface visualises the full set of Kubernetes resources, including services, pods, and ingress, as well as their current synchronisation and health status.
\begin{figure}[h]

In the local development environment, the Ingress resource may remain in a Progressing state despite successful application synchronisation. This behaviour is caused by the use of cloud-specific annotations (e.g. AWS ALB annotations) in combination with a local Kubernetes cluster (Docker Desktop), which does not provision an external load balancer. As a result, the status.loadBalancer field remains empty. This does not affect the internal functionality of the system, as services are still accessible via cluster networking and port-forwarding, and therefore does not indicate a deployment failure.

\subsection{Automated testing}

End-to-end smoke tests for the frontend are implemented with Playwright and run as part of the CI workflow on every push and pull request. The test suite covers six scenarios that verify the core user-facing behaviour of the application without requiring a live backend: all API calls are intercepted at the browser level using Playwright's route mocking, so tests run entirely within GitHub Actions without GPU or network access to the model services.

The scenarios covered are:

\begin{enumerate}
    \item \textbf{Page load.} The welcome screen and prompt text are visible on initial load.
    \item \textbf{Subject tabs.} All three subject tabs (FSD, FCS, DMA) are rendered.
    \item \textbf{Tab switching.} Switching to a different subject tab shows the welcome screen for that tab.
    \item \textbf{Chat flow.} Submitting a question via the input field displays the user message and the mocked assistant response in the conversation panel.
    \item \textbf{Source display.} After a response, expanding the Sources section renders the retrieved chunk filenames.
    \item \textbf{Tab isolation.} Each subject tab maintains an independent message history: switching away and back preserves the conversation without cross-tab contamination.
\end{enumerate}

Playwright reports are uploaded as a GitHub Actions artefact on every run and retained for seven days, allowing failures to be inspected without re-running the pipeline.

\subsection{Model versioning}

Each LoRA adapter is assigned a semantic version, which is recorded in subjects.yaml alongside its name and ChromaDB identifier. The version field is exposed at runtime through the /health endpoint, which returns a versions dictionary mapping each loaded subject to its adapter version. This allows operators to check which adapter version is running in production without inspecting the image or the file system directly. When an adapter is retrained, the version field is incremented, providing a simple audit trail in the Git history.
